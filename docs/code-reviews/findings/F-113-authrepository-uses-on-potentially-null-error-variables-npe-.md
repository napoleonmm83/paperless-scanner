# F-113 — AuthRepository uses !! on potentially-null error variables — NPE risk on login

**Severity:** `high` · **Area:** `data/error-handling` · **Effort:** `S`
**Issue:** https://github.com/napoleonmm83/paperless-scanner/issues/140

## Location

- **File:** `app/src/main/java/com/paperless/scanner/data/repository/AuthRepository.kt`
- **Lines:** `L55-L80`
- **Target:** `AuthRepository.detectServerProtocol`

## Problem

Five `!!` usages on `httpsError` / `httpError` inside when-branches. Variables are assigned only when an exception is caught; if neither call throws but a logical failure occurs, the variables stay null and the `!!` crashes the app at login time — a critical user path.

## Proposed Fix

Initialize errors with non-null fallback (`Exception("Unknown HTTP error")`) OR replace `!!` with elvis: `httpsError ?: someFallback`. Ensure each when-branch handles the null case explicitly. Add unit tests for all branches including the both-succeed-but-validation-fails scenario.

## Acceptance Criteria

- [ ] No `!!` operators in AuthRepository error handling
- [ ] Unit tests cover null-error branches
- [ ] No NPE possible at login
