# F-011 — SecureTokenStorage swallows exceptions, masking crypto failures

**Severity:** `medium` · **Area:** `security/keystore` · **Effort:** `M`
**Issue:** https://github.com/napoleonmm83/paperless-scanner/issues/37

## Location

- **File:** `app/src/main/java/com/paperless/scanner/data/datastore/SecureTokenStorage.kt`
- **Lines:** `L64-L90, L145-L171`
- **Target:** `SecureTokenStorage.getToken, SecureTokenStorage.saveToken`

## Problem

`getToken()` and `saveToken()` catch all exceptions and return null/false silently. Real failures (`AEADBadTagException`, permission denial, library bug) are indistinguishable from "no token yet". Recovery (`recoverCorruptedStorage`) deletes keystore + prefs non-atomically — partial failure leaves inconsistent state.

## Proposed Fix

Distinguish exception types: AEAD/corruption → `recoverAndRehydrate()`; PermissionDenied → log security event + propagate; transient → retry with backoff. Make recovery atomic (snapshot + restore on failure). Surface a `Result<T>` to TokenManager so UI can show "please re-login".

## Acceptance Criteria

- [ ] Exception taxonomy implemented (recoverable vs hard-fail vs retry)
- [ ] Recovery is atomic — partial failure is rolled back
- [ ] Result<T> bubbles to UI for non-recoverable cases
- [ ] Tests cover each taxonomy branch
