# F-034 — Crashlytics breadcrumbs scattered across upload paths with hardcoded strings

**Severity:** `low` · **Area:** `architecture/error-handling` · **Effort:** `S`
**Issue:** https://github.com/napoleonmm83/paperless-scanner/issues/60

## Location

- **File:** `app/src/main/java/com/paperless/scanner/data/repository/DocumentRepository.kt`
- **Lines:** `L76, L130, L157, L214-L240`
- **Target:** `DocumentRepository.uploadDocument, uploadMultiPageDocument`

## Problem

Breadcrumb labels ("UPLOAD_START", "UPLOAD_SUCCESS", "UPLOAD_ERROR") are hardcoded inline at multiple call sites. Refactoring upload code drifts breadcrumbs out of sync.

## Proposed Fix

Introduce `UploadMetricsTracker` with named methods (`logStart(pageCount)`, `logSuccess(taskId)`, `logError(errorType)`). Inject Crashlytics helper. Single source of truth for breadcrumb labels.

## Acceptance Criteria

- [ ] UploadMetricsTracker centralizes breadcrumb logging
- [ ] No hardcoded breadcrumb strings in DocumentRepository
- [ ] Tests verify tracker calls via mock
