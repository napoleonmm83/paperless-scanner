# F-003 — AppLock route reconstruction vulnerable to dual SavedStateHandle desync

**Severity:** `high` · **Area:** `security/applock` · **Effort:** `M`
**Issue:** https://github.com/napoleonmm83/paperless-scanner/issues/30

## Location

- **File:** `app/src/main/java/com/paperless/scanner/ui/navigation/AppLockNavigationInterceptor.kt`
- **Lines:** `L27-L67`
- **Target:** `AppLockNavigationInterceptor.reconstructRouteWithArgs`

## Problem

`reconstructRouteWithArgs` reads URIs from Navigation BackStackEntry SavedStateHandle, but ViewModel SavedStateHandle is the other half of CLAUDE.md's dual-sync pattern. If they desync, route reconstruction loses data on AppLock unlock OR Process Death recovery fails — depending which handle is stale.

## Proposed Fix

Introduce a `SavedStateBridge` helper that writes to both handles atomically on every URI/page change. Add debug-build assertion that detects desync at reconstruction time and logs it. Reference CLAUDE.md "AppLock & SavedStateHandle Pattern". Long-term: collapse to a single source of truth (ViewModel SavedStateHandle) and have the interceptor read via a stable accessor.

- [ ] Debug assertion catches desync at AppLock unlock
- [ ] Integration test: lock during scan, unlock — pages preserved
