# F-046 — UploadViewModel race: setDocumentUris vs reactive documentUrisStateFlow

**Severity:** `high` · **Area:** `architecture/reactive` · **Effort:** `M`
**Issue:** https://github.com/napoleonmm83/paperless-scanner/issues/70

## Location

- **File:** `app/src/main/java/com/paperless/scanner/ui/screens/upload/UploadViewModel.kt`
- **Lines:** `L75-L98, L104-L110`
- **Target:** `UploadViewModel.setDocumentUris, documentUrisStateFlow`

## Problem

Screen calls `setDocumentUris()` after VM creation, but reactive `documentUrisStateFlow` reads from SavedStateHandle. Order of operations means flow may emit empty list before manual set, racing dependent flows.

## Proposed Fix

Pass URIs into VM via assisted-injection / Hilt @AssistedFactory or read from BackStackEntry args inside the VM at init. Remove `setDocumentUris`. Alternatively expose a single source of truth `MutableStateFlow` written atomically with the SavedStateHandle.

## Acceptance Criteria

- [ ] URIs set during VM construction (no post-init mutator)
- [ ] No race between manual set and reactive read
- [ ] Tests verify URI availability before any dependent flow emits
