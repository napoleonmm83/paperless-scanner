# F-115 — BillingManager owns custom CoroutineScope without lifecycle cleanup

**Severity:** `medium` · **Area:** `performance/threading` · **Effort:** `S`
**Issue:** https://github.com/napoleonmm83/paperless-scanner/issues/142

## Location

- **File:** `app/src/main/java/com/paperless/scanner/data/billing/BillingManager.kt`
- **Lines:** `scope init`
- **Target:** `BillingManager.scope`

## Problem

`private val scope = CoroutineScope(SupervisorJob() + Dispatchers.Main)` lives for the singleton's lifetime with no `cancel()` exposed. Audited similar issues in `ServerHealthMonitor` (`pollingJob`) and `NetworkMonitor`. Long-lived scopes accumulate work + leak when the manager is recreated or the app process is recycled.

## Proposed Fix

Add `fun cancel()` (or override `Closeable.close()`) calling `scope.cancel()`. Tie to the consuming Activity/ViewModel lifecycle via Hilt or expose to `PaperlessApplication.onTerminate`. Long-term: prefer using `@ApplicationScope` injected `CoroutineScope` from a Hilt module so cleanup is centralized.

## Acceptance Criteria

- [ ] BillingManager exposes cancel() and is wired to lifecycle
- [ ] ServerHealthMonitor + NetworkMonitor reviewed similarly
- [ ] Leak canary (or test) confirms no leaked job after teardown
