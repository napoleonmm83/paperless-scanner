# F-006 — TokenManager *Sync() functions runBlocking from OkHttp interceptor — thread starvation risk

**Severity:** `medium` · **Area:** `security/tokens` · **Effort:** `L`
**Issue:** https://github.com/napoleonmm83/paperless-scanner/issues/33

## Location

- **File:** `app/src/main/java/com/paperless/scanner/data/datastore/TokenManager.kt`
- **Lines:** `L114-L137, L609-L734`
- **Target:** `TokenManager.getTokenSync, TokenManager.migrateTokenToSecureStorage`

## Problem

`*Sync()` functions wrap async DataStore calls in `runBlocking`. Called from `DynamicBaseUrlInterceptor`, they block OkHttp dispatcher threads. If DataStore IO contends or crypto stalls, the OkHttp thread pool can be starved.

## Proposed Fix

Cache server URL + token at app startup in an in-process volatile holder with synchronous read. Keep `runBlocking` only in true init/migration paths; never in request interceptors. Expose async `Flow<String?>` for UI. Annotate remaining sync APIs with `@WorkerThread`.

## Acceptance Criteria

- [ ] Server URL + token cached at startup, accessed without runBlocking from interceptors
- [ ] @WorkerThread annotation on remaining sync calls
- [ ] Stress test: rapid request bursts don't hang the dispatcher
- [ ] Tests still pass
