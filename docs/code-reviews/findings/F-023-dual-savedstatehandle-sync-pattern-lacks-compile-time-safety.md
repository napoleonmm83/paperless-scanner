# F-023 — Dual SavedStateHandle sync pattern lacks compile-time safety

**Severity:** `medium` · **Area:** `architecture/domain` · **Effort:** `L`
**Issue:** https://github.com/napoleonmm83/paperless-scanner/issues/49

## Location

- **File:** `app/src/main/java/com/paperless/scanner/ui/screens/scan/ScanScreen.kt`
- **Lines:** `manual sync sites in ScanScreen, BatchImportScreen, MultiPageUploadScreen`
- **Target:** `ScanScreen, BatchImportScreen, MultiPageUploadScreen`

## Problem

CLAUDE.md documents two SavedStateHandles (ViewModel + Navigation BackStackEntry) that must be kept manually in sync via `LaunchedEffect`. Any forgotten sync silently breaks AppLock OR Process Death recovery.

## Proposed Fix

Centralize behind a `rememberDualSavedState(viewModel, backStackEntry, key)` helper that asserts both writes happen atomically. Long-term consider a single source of truth + a stable accessor used by AppLockNavigationInterceptor.

## Acceptance Criteria

- [ ] rememberDualSavedState helper + tests
- [ ] All screens migrated to helper, no raw LaunchedEffect copies
- [ ] Process death + AppLock paths tested per screen
