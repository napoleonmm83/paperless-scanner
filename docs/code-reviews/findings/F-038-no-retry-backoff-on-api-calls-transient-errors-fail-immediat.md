# F-038 — No retry/backoff on API calls — transient errors fail immediately

**Severity:** `medium` · **Area:** `architecture/error-handling` · **Effort:** `M`
**Issue:** https://github.com/napoleonmm83/paperless-scanner/issues/64

## Location

- **File:** `app/src/main/java/com/paperless/scanner/data/repository/DocumentRepository.kt`
- **Lines:** `L118-L127, L202-L211, L675-L710, L800-L877, L920-L933`
- **Target:** `DocumentRepository.uploadDocument, uploadMultiPageDocument, downloadDocument, updateDocument, getDocumentHistory`

## Problem

API calls use plain try/catch with no exponential backoff or retry policy. Transient 5xx + timeout errors fail immediately. Large multi-page uploads have no resume support.

## Proposed Fix

Introduce `RetryPolicy` (max 3 retries, exponential backoff with jitter, retry only idempotent ops + 429/5xx). Wrap API calls via a helper `apiWithRetry { ... }`. For uploads consider partial-PDF resume.

## Acceptance Criteria

- [ ] RetryPolicy used for idempotent operations
- [ ] Non-idempotent ops not retried (avoid duplicate uploads)
- [ ] Tests simulate transient failures + verify retry count
