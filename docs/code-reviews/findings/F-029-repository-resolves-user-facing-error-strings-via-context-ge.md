# F-029 — Repository resolves user-facing error strings via Context.getString

**Severity:** `medium` · **Area:** `architecture/error-handling` · **Effort:** `M`
**Issue:** https://github.com/napoleonmm83/paperless-scanner/issues/55

## Location

- **File:** `app/src/main/java/com/paperless/scanner/data/repository/DocumentRepository.kt`
- **Lines:** `L136-L144, L222-L244, L353, L374`
- **Target:** `DocumentRepository.uploadDocument, uploadMultiPageDocument`

## Problem

Repository calls `context.getString(R.string.error_*)` directly. This couples data layer to Android resources, violates Clean Architecture, and breaks unit testability without Robolectric.

## Proposed Fix

Refactor `PaperlessException` to carry typed error codes only. Move string resolution to ViewModel/UI. Inject `@StringRes` IDs via the exception or sealed error type.

## Acceptance Criteria

- [ ] No context.getString call in DocumentRepository
- [ ] UI layer maps exceptions to localized strings
- [ ] Repository unit tests no longer require Context
