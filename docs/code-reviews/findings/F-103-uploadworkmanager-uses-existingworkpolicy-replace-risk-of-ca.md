# F-103 — UploadWorkManager uses ExistingWorkPolicy.REPLACE — risk of cancelled queues

**Severity:** `medium` · **Area:** `worker/idempotency` · **Effort:** `S`
**Issue:** https://github.com/napoleonmm83/paperless-scanner/issues/130

## Location

- **File:** `app/src/main/java/com/paperless/scanner/worker/UploadWorkManager.kt`
- **Lines:** `L34-L38`
- **Target:** `UploadWorkManager.scheduleUpload`

## Problem

Both schedulers use `REPLACE`. Rapid taps cancel prior work, which can drop items the worker had already begun queuing internally and confuses users.

## Proposed Fix

Switch to `KEEP` for the regular schedule. Provide an explicit "cancel and restart" UI for forced overrides. Add a UI debounce so rapid taps coalesce.

## Acceptance Criteria

- [ ] Schedule policy is KEEP by default
- [ ] Cancel-and-restart action available where needed
- [ ] Rapid tap test does not lose enqueued items
