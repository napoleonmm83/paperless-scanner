# F-010 — No certificate pinning for user-provided Paperless servers (MITM risk)

**Severity:** `high` · **Area:** `security/network` · **Effort:** `L`
**Issue:** https://github.com/napoleonmm83/paperless-scanner/issues/84

## Location

- **File:** `app/src/main/res/xml/network_security_config.xml`
- **Lines:** `L1-L8`

## Problem

The app talks to user-provided Paperless servers and trusts system + user CAs. No TLS pinning means a malicious CA / corporate proxy / on-path attacker can intercept tokens and document content over HTTPS.

## Proposed Fix

Add optional Trust-On-First-Use (TOFU) certificate pinning: capture cert SHA-256 on first successful login (showing fingerprint to user), persist in encrypted storage, and configure OkHttp `CertificatePinner`. Detect cert changes on subsequent connections and surface a warning UI requiring explicit re-trust.

## Acceptance Criteria

- [ ] TOFU pinning UI on first login (cert fingerprint displayed)
- [ ] Pinned hash stored encrypted; OkHttp CertificatePinner enforces it
- [ ] Cert change triggers a blocking warning dialog
- [ ] Tests verify pinning + change detection
